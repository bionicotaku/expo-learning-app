# 项目规范

## 1. 文档目标

本文档定义当前仓库的默认项目规范，用来约束后续实现的组织方式和最低质量标准。它覆盖：

- 目录与文件结构
- 分层与依赖方向
- 命名与导出方式
- 逻辑组织与状态管理
- MVP 阶段的 mock 数据实现方式
- 测试、验证与文档同步要求

这份规范不是通用模板，而是当前仓库的执行标准。后续新增模块、调整结构或扩展业务时，默认都应遵循本文档。

## 2. 适用范围与优先级

## 2.1 适用范围

本文档适用于：

- `src/` 下所有新增代码
- `docs/` 下所有新增设计文档
- 模块结构、调用关系、测试组织和文档同步方式

## 2.2 规则优先级

规则冲突时，按以下顺序处理：

1. `AGENTS.md`
2. [产品总览](./项目整体需求与应用概览.md)
3. [技术栈选型](./技术栈选型.md)
4. [网络层与事件上报规范](./网络层与事件上报规范.md)
5. 本文档
6. 当前仓库实际代码

其中：

- 产品目标、业务范围和 MVP 边界以上位产品文档为准
- 具体工程组织和日常实现方式以本文档为准
- 当前代码如果与文档不一致，后续改动时应以更高优先级文档为基准逐步收敛

## 3. 核心原则

所有后续实现默认遵循以下原则：

1. 先看产品与技术文档，再看当前实现，再动手写代码
2. 先搜索已有实现，不重复造轮子
3. 不为“可能的未来”提前设计重型抽象
4. 不保留兼容壳、旧路径和过渡实现
5. 模块职责清楚，依赖方向单向可解释
6. MVP 阶段虽然使用 mock 数据，但功能必须按真实异步业务流实现
7. 改结构就同步改文档，不允许代码和文档长期背离

## 4. 目标目录结构

当前项目的目标目录结构如下：

```text
src/
  app/                 Expo Router 路由与 layout
  pages/               页面级组合
  widgets/             页面级复合视图
  features/            用户动作与业务能力
  entities/            领域实体、类型、领域数据读写
  shared/              通用基础设施
```

### 4.1 路由层

`src/app/` 只允许放：

- route 文件
- `_layout.tsx`
- route group
- 与路由本身强相关的最小入口适配

不允许把以下内容直接放到 `src/app/`：

- 业务组件
- 业务 hooks
- 页面状态逻辑
- mock 数据
- SDK 客户端初始化

### 4.2 页面层

`src/pages/` 负责：

- 单个页面的组合入口
- 页面所需 widgets / features 的装配
- 页面级 provider 的最小拼装

`pages` 不负责：

- 直接访问远程服务
- 直接创建第三方 SDK client
- 宽泛的通用组件沉淀

### 4.3 Widget 层

`src/widgets/` 负责：

- 页面中占较大可复用区域的复合视图
- 多个 feature / entity / shared 能力的组合
- 页面内较重的视图编排

`widgets` 不负责：

- 直接承载全局路由逻辑
- 直接初始化应用级状态容器

### 4.4 Feature 层

`src/features/` 负责：

- 用户动作
- 业务交互
- 写操作
- 派生状态
- 与单一业务能力相关的读取编排

例子包括：

- 认证
- feed 分页
- 播放控制
- 收藏
- 历史
- 沉淀条目保存

### 4.5 Entity 层

`src/entities/` 负责：

- 领域类型
- 领域对象
- 领域级读写接口
- mock 数据生成或实体映射
- 不依赖页面上下文的领域逻辑

entity 不应直接知道页面结构，也不应承载跨多个业务动作的 orchestration。

### 4.6 Shared 层

`src/shared/` 负责：

- 通用 UI 原子
- SDK client
- 工具函数
- 主题
- 配置
- 通用 hooks
- 与具体业务无关的底层能力

推荐继续向下拆分为：

```text
src/shared/
  api/
  config/
  lib/
  theme/
  ui/
  types/
```

当前未全部建立，不代表后续可以把通用代码随意堆回 `components/` 或 `hooks/`。

## 5. 当前仓库的过渡目录处理

当前仓库仍存在一些 Expo 模板遗留目录：

- `src/components`
- `src/constants`
- `src/hooks`

这些目录当前只视为过渡遗留，不视为后续标准结构。

规则如下：

- 不在这些目录中新增业务代码
- 不再把新的通用组件放入 `src/components`
- 不再把新的主题常量放入 `src/constants`
- 不再把新的通用 hook 放入 `src/hooks`
- 修改这些目录中的文件时，优先评估是否顺手迁移到 `shared/` 或具体业务 slice

除非是专门的迁移任务，否则可以暂时保留它们，但不能继续扩散使用。

## 6. 层级依赖规则

默认依赖方向如下：

```text
app -> pages, shared
pages -> widgets, features, entities, shared
widgets -> features, entities, shared
features -> entities, shared
entities -> shared
shared -> shared
```

### 6.1 默认禁止的依赖

- `shared` 反向依赖 `entities / features / widgets / pages / app`
- `entities` 依赖 `features / widgets / pages / app`
- `features` 依赖 `widgets / pages / app`
- `widgets` 依赖 `pages / app`
- `pages` 互相依赖

### 6.2 同层依赖规则

默认不鼓励跨同层 slice 强耦合。

具体规则：

- `features` 默认不直接依赖其他 `features`
- `widgets` 默认不直接依赖其他 `widgets`
- `entities` 默认不直接依赖其他 `entities` 的私有实现

如果确实需要跨同层协作，优先做两件事：

1. 先判断职责是否拆错
2. 如确有必要，只通过对方的公共导出访问

## 7. Slice 内部结构规范

每个 slice 默认只保留必要子目录，不为整齐而整齐。

推荐的最小结构是：

```text
<slice>/
  index.ts
  model/
  api/
  ui/
  lib/
```

说明：

- `model/`：状态、类型、纯逻辑、reducer、selectors、业务规则
- `api/`：读写接口、mock repository、远程请求封装
- `ui/`：组件、视图壳、局部交互 UI
- `lib/`：仅属于本 slice 的辅助函数
- `index.ts`：slice 的公共导出面

不是每个 slice 都必须具备全部子目录。没有的就不要建空目录。

## 8. 公共导出与导入规范

### 8.1 切片公共 API

跨 slice 导入时，默认通过切片根级 `index.ts` 进入。

例如允许：

```ts
import { VideoFeed } from "@/widgets/video-feed";
import { createFeedPlaybackState } from "@/features/video-playback";
import { isFeedVideoItem } from "@/entities/video";
```

不推荐：

```ts
import { isFeedVideoItem } from "@/entities/video/model/types";
```

### 8.2 何时允许深层导入

只有两种情况允许深层导入：

- slice 内部文件之间的实现导入
- 明确不想暴露到公共 API 的测试辅助导入

一旦某个能力需要被多个外部调用，就应提升到 slice 根导出。

### 8.3 路径别名

跨目录导入统一使用 `@/` 别名，不使用长相对路径。

例如：

```ts
import { colors } from "@/shared/theme/colors";
```

不推荐：

```ts
import { colors } from "../../../shared/theme/colors";
```

## 9. 文件命名与导出规范

### 9.1 文件命名

- 普通文件统一使用 `kebab-case`
- React 组件文件也使用 `kebab-case`
- route 文件遵循 Expo Router 约定
- 测试文件使用 `<name>.test.ts` 或 `<name>.test.tsx`

### 9.2 导出规范

- route 组件默认使用 `default export`
- 非 route 模块默认使用 `named export`
- 类型优先使用 `export type`
- 切片入口通过 `index.ts` 聚合对外暴露内容

### 9.3 命名规范

- hook 必须以 `use-` 对应的 `useXxx` 命名
- reducer / state / action 命名要体现业务含义，不允许出现无语义的 `data`, `info`, `helper`
- 文件名优先表达业务对象，不用 `utils.ts`、`helpers.ts`、`common.ts` 这类大杂烩命名

允许存在局部辅助文件，但名称必须具体，例如：

- `pagination-helpers.ts`
- `session-storage.ts`
- `video-item-mapper.ts`

## 10. 路由组织规范

后续路由结构默认按以下模式展开：

```text
src/app/
  _layout.tsx
  (auth)/
  (tabs)/
  modal/
```

默认约束：

- 每个 route group 用 `_layout.tsx` 管自己的导航容器
- 认证区和主应用区必须分组
- Tabs 只承担主导航职责，不在 tab layout 中塞业务状态逻辑
- 业务页面本体继续由 `pages/` 提供

route 文件职责应尽量薄：

- 从 `pages/` 引入页面入口
- 定义局部 `Stack.Screen` 配置
- 不承载大段业务逻辑

## 11. 逻辑组织规范

## 11.1 页面与视图层

- 页面组件负责组合，不直接写业务数据读写
- widget 负责复合视图，不直接初始化全局服务
- UI 组件优先接受已整理好的 props，不把多段业务逻辑塞进视图渲染函数

## 11.2 业务能力层

每个业务能力应尽量满足：

- 读取逻辑有清楚入口
- 写入逻辑有清楚入口
- 状态来源单一
- 副作用位置集中

例如：

- 认证逻辑集中在 `features/auth`
- feed 分页集中在 `features/feed-pagination`
- 播放状态集中在 `features/video-playback`

## 11.3 副作用规则

副作用默认放在以下位置：

- React Query query / mutation
- feature 层 hook
- provider
- route 切换相关的最小入口层

不允许把副作用随意散落在多个页面组件中重复实现。

## 11.4 状态管理规则

当前默认组合是：

- 远程或服务型异步状态：React Query
- 认证与少量应用级状态：Context
- 局部复杂交互状态：`useReducer`
- 轻量局部状态：`useState`

当前不把 Zustand 视为默认标准。后续若引入，必须先证明：

- 它解决的是跨页面纯客户端状态问题
- 它不能更合理地放进 React Query、Context 或局部 reducer

## 12. MVP 阶段的 mock 数据规范

MVP 阶段除 `Supabase Auth` 外，业务数据全部使用 mock 数据，但实现方式必须接近真实系统。

### 12.1 必须满足的要求

- 所有读取与写入都通过明确的异步 API 暴露
- 页面只消费 query / mutation / action，不直接读写页面内常量
- 页面必须具备 loading、empty、error、success、mutation in-flight 状态
- 操作必须真的改变状态，而不是只改文案或只弹提示

### 12.2 推荐组织方式

mock 数据优先放在：

- `entities/<name>/api`
- `features/<name>/api`
- `features/<name>/model/mock-*.ts`

取决于它更像：

- 实体级读写
- 业务动作级读写

### 12.3 后续替换规则

后续接真实服务时，应尽量保持：

- 调用入口不变
- 返回结构不变或只做小范围适配
- 页面层不重写

替换目标应是数据源，而不是整块 UI 与状态模型。

## 13. 认证相关特殊规则

认证在 MVP 阶段只允许耦合到 Supabase Auth，但必须集中收口。

默认约束：

- 只有 `shared/lib/supabase` 负责创建 client
- 只有 `features/auth` 直接调用 `supabase.auth.*`
- 页面、widget、其他 feature 不直接调用 Supabase SDK

如果未来认证供应商变更，重构面应尽量集中在认证切片内部。

## 14. 测试规范

## 14.1 测试分层

当前仓库默认采用三层测试思路：

- 纯逻辑测试
- 业务切片测试
- 启动级验证

### 纯逻辑测试

适用于：

- reducer
- helper
- mapper
- mock feed 生成
- 播放窗口判定

这类测试必须 colocate 在实现附近。

### 业务切片测试

适用于：

- mock API 读写行为
- query / mutation 的状态流转
- slice 的关键业务规则

优先验证行为，不做无意义快照。

### 启动级验证

适用于：

- Expo 页面
- 路由切换
- 原生视频播放行为
- provider 初始化

这类验证通过 `npm run ios` 完成基础启动核对。

## 14.2 测试文件位置

默认规则：

- 单元测试 colocate 到源文件旁边
- 文件名为 `<source-name>.test.ts` 或 `<source-name>.test.tsx`

例如：

```text
src/features/feed-pagination/model/mock-feed.ts
src/features/feed-pagination/model/mock-feed.test.ts
```

当前不建立全局 `tests/` 大仓库，也不把所有测试集中到根目录。

## 14.3 新增代码的最低测试要求

新增模块时，至少满足以下标准：

- 纯逻辑文件必须有单元测试
- mock 数据读写逻辑必须有测试
- reducer / selector / mapper 必须有测试
- 涉及分页、保存、删除、切换这类关键交互的逻辑必须有测试

以下内容可以暂时不强求单测，但仍需通过运行验证：

- 纯视觉排版
- 只做简单装配的 route 文件
- 极薄的页面壳组件

## 14.4 禁止的测试方式

- 只靠 snapshot 证明业务逻辑正确
- 只测 happy path，不测边界和错误态
- 把测试写成实现细节断言，导致一改内部结构就全部失效

## 14.5 当前回归命令

默认回归命令是：

- `npm run quick-check`
- `npm run check`

规则如下：

- 日常编码快速回归先跑 `npm run quick-check`
- 标准最终验收跑 `npm run check`
- 只改文档可以不跑代码检查
- 影响 Expo 页面、路由或原生播放行为时，额外补一次 `npm run ios`

## 15. 文档规范

### 15.1 文档位置

- 设计与规范文档放在 `docs/`
- 目录级实现说明放在对应目录的 `README.md`
- 临时过程文档放在 `docs/temp/`

### 15.2 必须同步更新文档的情况

如果改动影响以下任一内容，必须同步更新相关文档：

- 目录结构
- 模块职责
- 依赖方向
- 数据流
- mock / live 数据策略
- 测试方式
- 维护方式

### 15.3 新目录的 README 要求

出现新的业务目录或明显自定义结构时，应补一个目录级 `README.md`，至少写清：

- 目录结构
- 模块职责
- 边界
- 主要依赖
- 主要调用链

如果偏离本文档，必须在对应 `README.md` 中明确说明：

- 为什么偏离
- 偏离了哪条规则
- 边界和维护约束是什么

## 16. 明确禁止的模式

以下模式默认禁止：

- 在 `src/app` 中编写业务实现
- 在页面组件中直接初始化 SDK client
- 在多个页面里复制同一套业务读取 / 写入逻辑
- 把 mock 数据常量直接塞进页面 JSX
- 新增 `utils.ts` / `helpers.ts` / `common.ts` 大杂烩文件
- 新增代码继续堆到 `src/components` / `src/constants` / `src/hooks`
- 跨层反向依赖
- 为兼容旧结构保留重复壳层
- 没有同步文档的结构性改动

## 17. 新功能落地检查清单

开始实现一个新模块前，默认先检查：

1. 它属于 `page / widget / feature / entity / shared` 的哪一层
2. 它的公共入口应该暴露在哪里
3. 它是否需要 mock API / query / mutation
4. 它的 loading / error / empty / success 状态是否定义清楚
5. 哪些逻辑必须写测试
6. 是否需要新增目录级 `README.md`
7. 是否会影响现有文档

如果这七件事说不清楚，就不应该直接开始实现。

## 18. 总结

这份规范的目的不是增加流程，而是让后续实现遵循同一套边界：

- 结构统一
- 依赖方向统一
- 数据策略统一
- 测试要求统一
- 文档同步方式统一

这样做的结果是：MVP 阶段即使大量使用 mock 数据，仓库也仍然会像一个真实项目那样逐步生长，而不是变成一个堆满页面壳和临时代码的演示仓库。
