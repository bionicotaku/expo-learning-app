# 网络层与事件上报规范

## 1. 文档目标

本文档定义当前项目的网络层标准，用来回答以下问题：

- 前端请求按什么类型拆分
- 每一类请求由谁发起、如何去重、如何重试、是否允许持久化
- React Query、Supabase Auth、mock repository、未来真实后端之间如何分工
- 后续接入真实内容服务时，应该替换哪一层，不应该重写哪一层

这份文档服务于当前仓库，不是通用前端架构模板。产品范围见 [项目整体需求与应用概览](./项目整体需求与应用概览.md)，技术栈基线见 [技术栈选型](./技术栈选型.md)，工程组织规则见 [项目规范](./项目规范.md)。

## 2. 适用范围与优先级

本文档适用于：

- `src/shared/api`
- `src/shared/telemetry`
- `src/entities/*/api`
- `src/features/*/api`
- 所有通过 query / mutation / action 发起的异步业务读写

规则冲突时，默认优先级如下：

1. `AGENTS.md`
2. [项目整体需求与应用概览](./项目整体需求与应用概览.md)
3. [技术栈选型](./技术栈选型.md)
4. 本文档
5. [项目规范](./项目规范.md)
6. 当前仓库实际实现

## 3. 核心结论

当前项目的网络与异步读写，统一按三类处理：

1. `interactive-read`
2. `interactive-write`
3. `telemetry`

这三类不是按 HTTP 方法区分，而是按业务语义区分。

### 3.1 `interactive-read`

指用户正在等待内容返回给自己看的读取行为。

典型例子：

- feed 列表
- 视频详情
- 收藏列表
- 历史记录列表
- 沉淀区列表

它的目标是：

- 快
- 准
- 可取消陈旧请求
- 可缓存
- 失败后可见并可重试

### 3.2 `interactive-write`

指用户明确触发、并期望应用状态真正发生变化的写入行为。

典型例子：

- 收藏 / 取消收藏
- 保存学习条目
- 删除学习条目
- 修改条目标记
- 提交笔记

它的目标是：

- 尽快反馈
- 明确成功或失败
- 控制副作用
- 必要时可乐观更新

### 3.3 `telemetry`

指用户不直接等待结果、允许延迟、允许部分失败的行为与状态上报。

典型例子：

- 视频观看时长
- 最后观看进度
- 暂停 / 恢复 / 滑走事件
- 页面曝光
- 弱一致统计事件

它的目标是：

- 不阻塞主交互
- 可批量
- 可合并
- 可重试
- 必要时可本地持久化

## 4. 必须区分的几个概念

### 4.1 `query key`

`query key` 是 React Query 用来标识“这份读数据是谁”的键。

例如当前 feed 已经采用：

```ts
['feed', 'main']
```

它的作用是：

- 标识数据身份
- 管理缓存
- 让同语义读取共享结果
- 让失效、刷新、分页归属于同一份服务数据

它不是：

- HTTP header
- 幂等键
- 服务端请求唯一 ID

### 4.2 `in-flight`

`in-flight` 表示请求已经发出，但尚未返回。

在当前项目里，常见形态包括：

- `isPending`
- `isFetching`
- `isFetchingNextPage`
- `isPending` mutation

它的作用是：

- 阻止重复触发
- 控制 loading / disabled 状态
- 帮助区分“还没发”和“已经在路上”

### 4.3 `pending operation key`

这是写操作侧的本地去重键，用来标识“某个资源上的某个动作是不是已经在执行中”。

例如：

- `favorite:video-123`
- `save-clip:clip-42`
- `delete-note:note-9`

它通常存在于前端本地状态，不发送到服务端。

### 4.4 `idempotency key`

这是发送给服务端的幂等键，用来表达：

> 这几次请求是同一个用户意图的重复投递，请服务端只算一次。

它只适用于特定写操作，不是所有请求的默认标准。

### 4.5 持久化队列

持久化队列指的是：

- 请求或事件先写入本地存储
- 即使 App 被杀后重新启动，也仍可恢复并继续发送

当前项目里，默认只有 `telemetry` 允许采用这一策略。

## 5. 项目级分层与边界

## 5.1 当前推荐结构

```text
src/shared/
  api/                  通用请求基础层
  telemetry/            事件队列、flush、批量发送
  lib/supabase/         Supabase client

src/entities/<name>/api/
  ...                   领域读写、mock repository、未来真实 repository

src/features/<name>/api/
  ...                   业务动作级写入或特殊读取编排
```

说明：

- `shared/api` 负责通用传输问题，不负责业务语义
- `entities/*/api` 负责领域级读写，例如 feed page、收藏列表、历史列表
- `features/*/api` 负责动作级写入，例如收藏、取消收藏、保存条目
- `shared/telemetry` 独立于普通 query / mutation 通道

## 5.2 页面和 widget 的边界

页面和 widget 不直接写：

- `fetch(...)`
- `supabase.auth.*`
- 任意 ad-hoc 重试逻辑
- 任意 ad-hoc 队列逻辑

页面和 widget 只消费：

- query
- mutation
- feature action
- controller 暴露出的稳定状态与回调

## 5.3 Auth 的特殊地位

认证是当前项目里的特殊通道：

- MVP 阶段只允许使用 Supabase Auth
- 只有 `shared/lib/supabase` 负责 client
- 只有 `features/auth` 直接调用 `supabase.auth.*`

也就是说：

- Auth 在业务语义上仍然属于读写行为
- 但在传输实现上，它不走通用业务 HTTP 通道

不要为了“统一”把 Supabase Auth 硬塞进普通 `request()` 抽象里。

## 6. 通用请求基础层规则

## 6.1 传输实现

当前推荐：

- 优先使用 `fetch`
- 如后续需要更贴近 Expo 推荐路径，可引入 `expo/fetch`
- 不引入 `axios`

原因：

- 当前项目规模不需要为了拦截器生态引入额外复杂度
- Expo networking guidance 也更倾向于 `fetch` 路线

## 6.2 `shared/api` 应负责什么

当前 `src/shared/api` 应只负责：

- `baseUrl` 组装
- 通用 headers
- token 注入
- 超时控制
- `AbortSignal`
- JSON encode / decode
- 错误归一化

它不负责：

- 业务 query key
- feed 分页逻辑
- 收藏动作编排
- telemetry 队列
- Supabase Auth 细节

## 6.3 通用错误模型

网络基础层最终应统一抛出项目级错误对象，例如：

- `status`
- `code`
- `message`
- `retryable`

目的不是做过度封装，而是避免每个 slice 重新解释原始网络错误。

## 6.4 全局重试基线

当前仓库已经把 React Query 全局默认设置为：

- `queries.retry = false`
- `mutations.retry = false`

这代表当前基线是：

- 不自动假设超时可安全重试
- 每个高价值请求类别单独判断是否应该 override

这是正确的默认值，后续不应轻易改成全局自动重试。

## 7. `interactive-read` 规范

## 7.1 适用范围

凡是“用户正在等内容回来展示”的读取行为，默认都按 `interactive-read` 处理。

包括：

- feed
- 视频详情
- 收藏列表
- 历史列表
- 沉淀区列表

## 7.2 标准实现

默认实现方式：

- `useQuery`
- `useInfiniteQuery`

读取不应先写成手工 `useEffect + fetch + useState`，再在多个页面里重复实现。

## 7.3 query key 规则

所有读取都必须有稳定、可解释的 `queryKey`。

规则如下：

1. 第一段写领域名
2. 第二段写资源或场景名
3. 后续再带影响结果的参数

例如：

```ts
['feed', 'main']
['feed', 'main', { filter: 'favorites' }]
['video', videoId]
['collection', 'saved']
['history', userId]
```

禁止：

- 使用随机值作为 `queryKey`
- 使用每次 render 都新建但语义相同的不可稳定对象
- 把 mutation 的动作状态混进 query key

## 7.4 去重机制

`interactive-read` 的去重，依赖两层：

1. `queryKey` 去重同语义请求
2. `in-flight` 状态阻止重复触发

以当前 feed 为例：

- `['feed', 'main']` 代表主 feed
- 同 key 的 render 不应重复发第一页请求
- `isFetchingNextPage` 为 `true` 时，不应再次触发 `fetchNextPage()`

## 7.5 缓存策略

读取允许缓存，且应默认由 React Query 管理。

但要明确区分：

- 缓存已拿到的数据
- 持久化待发送请求

当前标准是：

- 可以缓存 query 结果
- 不持久化读取请求队列

未来若确实需要离线读体验，可以讨论持久化 query cache，但那属于后续数据策略增强，不是当前标准。

## 7.6 取消与陈旧请求

对于可能因为参数变化而失效的读请求，应优先支持：

- 参数变化时按新 `queryKey` 重新归类
- 需要时取消旧请求
- 避免陈旧结果覆盖当前场景

feed 的分页预取不需要复杂取消机制，但筛选式读取、详情跳转读取、搜索式读取后续需要遵守这个原则。

## 7.7 超时与重试

当前默认：

- 不自动重试
- 失败后展示 error
- 提供显式 retry 入口

只有满足以下条件时，才可以对单个读取增加有限自动重试：

- 请求无副作用
- 用户当前不依赖“立即失败反馈”做决策
- 重试不会显著拖长核心交互等待

即便如此，也应限制为少量重试，而不是无限重试。

## 7.8 持久化规则

`interactive-read`：

- 不持久化待发送请求
- 不做重启恢复未完成请求

App 被杀后，下一次进入页面重新发请求是正确行为。

## 7.9 UI 状态要求

读取页面至少要具备：

- loading
- empty
- error
- success

分页读取还必须具备：

- initial loading
- append loading
- append error 或 retry

## 8. `interactive-write` 规范

## 8.1 适用范围

凡是“用户明确点了一下，希望应用状态真的改变”的行为，默认都按 `interactive-write` 处理。

包括：

- 收藏 / 取消收藏
- 保存条目 / 删除条目
- 更新复习标记
- 提交笔记

## 8.2 标准实现

默认实现方式：

- `useMutation`
- feature action + mutation

不要在组件里直接：

- 写裸 `fetch`
- 写 ad-hoc 的 pending 布尔值
- 把副作用散落在多个页面中

## 8.3 接口语义规则

写接口优先设计成“设置目标状态”，而不是“切换状态”。

优先：

- `setFavorite(videoId, true)`
- `setFavorite(videoId, false)`
- `PUT /favorites/:id`
- `DELETE /favorites/:id`

谨慎使用：

- `toggleFavorite(videoId)`

原因是：

- `toggle` 在重复请求、超时重发、并发点击时语义不稳定
- 设置目标状态更接近幂等语义

## 8.4 本地去重机制

`interactive-write` 默认不靠 `queryKey` 去重，而是靠：

- 本地 pending 状态
- 资源级 operation key

例如：

- `favorite:video-123`
- `save-card:clip-88`

规则如下：

- 同一资源的同一动作 pending 时，不再重复触发
- pending 时按钮可禁用，或同动作被本地吞掉
- 不因多次点击而并发发出多个等价写请求

## 8.5 幂等键规则

不是每个写请求都要发送幂等键。

只有满足以下任一条件时，才要求幂等键：

1. 请求是非幂等创建型写入
2. 请求可能自动重试
3. 请求可能离线补发
4. 重复投递会产生重复实体或重复副作用

典型需要幂等键的场景：

- 创建评论
- 创建笔记实体
- 提交订单
- 任何命令型 `POST create`

典型不必默认带幂等键的场景：

- `PUT favorite`
- `DELETE favorite`
- 明确设置目标状态的更新接口

注意：

- 幂等键代表“同一个用户意图”
- 重试同一次动作时必须复用同一个幂等键
- 新的用户动作必须生成新的幂等键

不要把“每个 HTTP 请求都附带一个新随机 key”误当成幂等设计。

## 8.6 超时与重试

`interactive-write` 默认：

- 不自动重试
- 超时不自动重传
- 由 UI 明确失败并允许用户重试

原因是：

- 客户端超时不代表服务端一定没处理
- 自动重传可能导致重复副作用

只有在以下条件全部满足时，才可做有限自动重试：

- 接口本身已幂等化
- 服务端能正确处理重复投递
- 前端不会因此制造语义混乱

## 8.7 乐观更新

`interactive-write` 可以做乐观更新，但必须满足：

1. 回滚逻辑明确
2. 失败后用户可感知
3. 不把乐观态误当最终真相

像收藏这类切换可见状态的动作，适合乐观更新。  
像创建复杂实体、服务端可能补全字段的创建动作，需更谨慎。

## 8.8 与 query 的关系

写操作成功后，应按需做：

- 直接更新相关 query cache
- 或失效相关 query

例如收藏成功后，可能需要：

- 更新当前视频的收藏态
- 失效收藏列表
- 失效相关详情 query

不要把 mutation 成功后的同步责任留给页面层手工拼凑。

## 8.9 持久化规则

`interactive-write` 默认：

- 不进入持久化请求队列
- 不在 App 重启后自动补发

只有明确要支持“离线写入补发”时，才可以单独为某类写操作设计补发机制。  
这不属于当前 MVP 默认能力。

## 9. `telemetry` 规范

## 9.1 适用范围

凡是用户不直接等待结果、且允许延迟与容忍失败的事件，都按 `telemetry` 处理。

包括：

- 视频观看时长
- 最后观看进度
- 页面曝光
- 播放器状态切换事件
- 滑走事件

## 9.2 不走 React Query

`telemetry` 不应建模为普通 query / mutation。

原因：

- 量大
- 发送频率高
- 不需要逐条用户可见反馈
- 更适合合并、批量发送和后台 flush

因此它应走独立通道，例如：

```text
src/shared/telemetry/
  model/
  api/
  lib/
```

## 9.3 队列与持久化

`telemetry` 默认允许：

- 本地内存队列
- 本地持久化队列
- App 重启后恢复发送

这正是它和前两类的本质差别。

当前仓库已经建立了 `src/shared/telemetry` 的目录和类型落点，但尚未实现队列、flush 和持久化。

## 9.4 批量与合并

`telemetry` 不应逐条即时发送。

推荐策略：

- 同类事件合并
- 定时 flush
- 达到批量阈值 flush
- 生命周期触发 flush

尤其是视频进度类事件，应尽量做聚合：

- `watchDuration`：本地累计增量
- `lastProgress`：只保留最新值
- `viewEvent`：必要时合并为批量包

## 9.5 flush 时机

推荐触发时机：

- 达到事件数阈值
- 达到时间阈值
- 视频切换
- 播放暂停
- App 进入后台
- 页面离开

不要把 flush 时机绑死在某一个页面组件上。

## 9.6 重试与退避

`telemetry` 允许：

- 超时后自动重试
- 指数退避
- 后续再次 flush

但必须有上限：

- 最大重试次数
- 最大队列长度
- 最大单批大小

## 9.7 丢弃策略

`telemetry` 不要求无限保真。

所以必须预先定义：

- 队列满了时丢弃最旧还是最新
- 哪类事件可被覆盖
- 哪类事件必须优先保留

例如：

- `lastProgress` 可覆盖旧值
- `watchDuration` 可聚合累计
- 高频弱价值曝光事件可被裁剪

## 9.8 失败反馈

`telemetry` 失败默认不弹用户级错误。

它的结果应主要用于：

- 开发日志
- 调试面板
- 后台监控

而不是打断主交互。

## 10. 三类请求对照表

| 维度 | interactive-read | interactive-write | telemetry |
| --- | --- | --- | --- |
| 用户是否直接等待结果 | 是 | 是 | 否 |
| 典型实现 | `useQuery` / `useInfiniteQuery` | `useMutation` | 队列 + flush |
| 主去重手段 | `queryKey` + `in-flight` | pending operation key | 队列聚合 |
| 是否默认自动重试 | 否 | 否 | 是，可退避 |
| 是否默认持久化队列 | 否 | 否 | 是，可持久化 |
| 是否需要幂等键 | 否 | 视写入语义而定 | 通常不靠幂等键，靠事件聚合 |
| 是否允许乐观更新 | 不适用 | 视动作而定 | 不适用 |
| 用户失败反馈 | 显示 error / retry | 明确失败反馈 | 默认不打断用户 |

## 11. 当前项目的直接映射

## 11.1 feed

当前 feed 已经属于 `interactive-read`，并采用了正确方向：

- 使用 `useInfiniteQuery`
- 使用稳定 `queryKey`
- 使用 `isFetchingNextPage` 控制重复分页触发
- 全局默认不自动重试

这部分后续接真实 feed 服务时，原则上只替换 repository，不重写页面模型。

## 11.2 收藏

收藏尚未正式实现，但默认应按 `interactive-write` 设计：

- `useMutation`
- 设置目标状态优先于 `toggle`
- 本地 pending 去重
- 成功后失效或更新相关 query

## 11.3 观看进度与播放事件

这部分目前尚未建立完整通道，后续应直接按 `telemetry` 路线实现，不应先写成：

- 页面里零散 `fetch`
- 每次进度变化立刻单发请求
- 普通 mutation 列表

## 12. 测试与验证要求

## 12.1 `interactive-read`

至少覆盖：

- query key 稳定性
- 同 key 请求状态流转
- 分页追加规则
- in-flight guard
- loading / empty / error / success

## 12.2 `interactive-write`

至少覆盖：

- pending 去重
- 成功与失败状态流转
- 乐观更新与回滚
- 相关 query 失效或 cache 更新

## 12.3 `telemetry`

至少覆盖：

- 入队
- 聚合
- 批量 flush
- 重试与退避
- 队列恢复
- 容量上限与丢弃策略

## 13. 分阶段落地建议

### 第一阶段

先保持当前 feed 继续按 `interactive-read` 路线演进。

### 第二阶段

在收藏、历史、沉淀区等真实写操作开始实现时，补齐：

- `shared/api`
- `features/<name>/api`
- mutation 规范落地

### 第三阶段

在观看进度、曝光、行为统计开始落地时，再建立：

- `shared/telemetry`
- 队列
- flush
- 批量发送
- 持久化恢复

## 14. 总结

当前项目的网络层不是“统一大协议栈”，而是“一个轻量请求基础层 + 三类语义化请求通道”。

最重要的标准不是把所有请求塞进同一套机制，而是：

- 读取按读取处理
- 写入按写入处理
- 事件按事件处理

如果这三类边界保持清楚，后续从 mock 数据切换到真实服务时，替换的将主要是数据源和底层传输，而不是整套页面状态与交互模型。
