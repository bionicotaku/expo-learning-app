# 观看事件上报设计文档

本文描述当前移动端观看事件上报设计。当前已落地的是 fullscreen watch-progress 上报；未来学习互动事件可以复用同一个 active video visit 关联 ID，但必须由独立 feature、独立 queue、独立 API 契约维护。

## 1. 目标

watch-progress 的目标不是记录播放器每一帧变化，而是以低频、可合并、可延迟的方式，把一次视频观看 session 的最新摘要同步给后端：

- 当前播放到哪里：`position_ms`
- 本次 active visit 内真实观看了多久：`active_watch_ms`
- 这次观看属于哪个视频和播放会话：`video_id`、`watch_session_id`
- 上报发生在哪个客户端环境和页面表面：`client_context`、`source_surface`

这类请求属于 `telemetry`，不是用户等待结果的 interactive request。失败不打断播放，不弹用户级 toast，后续 flush 可以重试。

## 2. 非目标

当前 watch-progress MVP 不做这些事情：

- 不逐条发送高频 `timeUpdate`
- 不把每次 pause、resume、seek、字幕点击都作为 watch-progress 请求发送
- 不做 batch HTTP API
- 不做本地离线持久化
- 不接 AppState 后台 flush
- 不把学习互动事件塞进 watch-progress body
- 不由播放器组件直接 import API repository
- 不由前端上报 `is_completed`、`duration_ms`、`watch_ratio` 或 `user_id`

这些边界是为了避免 watch-progress 变成混杂的 clickstream，也避免 UI 层承担上报协议职责。

## 3. 模块边界

### 3.1 active video visit 层

`src/widgets/fullscreen-video-pager/model/use-fullscreen-playback-session.ts` 持有 active video visit 的身份。

当前约定：

- 首次 active video commit 时生成 `watchSessionId`
- 真实 active video 变化时生成新的 `watchSessionId`
- 切回同一个视频也是新的 active visit
- pause、resume、seek、弹窗、字幕点击不生成新 ID
- `getRowRenderState(videoId, index)` 只给当前 active row 返回 `watchSessionId`
- 非 active row 的 `watchSessionId` 固定为 `null`

`watchSessionId` 的所有权在 active video visit 层，不在 `features/video-watch-progress`。这样未来学习互动事件可以和观看进度共享同一个 visit 关联 ID。

### 3.2 fullscreen session / pager / row 层

`FullscreenVideoRow` 只从 active row 把播放事实转交给 pager：

- `videoId`
- `watchSessionId`
- `playbackRate`
- `progressSnapshot`

`FullscreenVideoSession` 持有 `useVideoWatchProgressReporter()` 和 watch-progress flush 生命周期。它接收 pager 转发的 active row progress sample，再交给 reporter。它同时读取 video screen focus 状态，并把 focus 作为播放、采样和定时 flush 的生命周期 gate。

`FullscreenVideoPager` 只负责 active row gating 和 sample 转发。pager 不持有 reporter，不计算 `active_watch_ms`，不拼 API body，不判断服务端完成状态，不 import `Platform`，不直接调用 watch-progress repository。screen unfocused 时，pager 不再把 active row progress sample 转发给 session。

### 3.3 video-watch-progress feature

`src/features/video-watch-progress` 是当前 watch-progress API 和 runtime reporter 的 owner。

职责：

- 定义 watch-progress request body 类型
- 暴露 `reportVideoWatchProgress(body)`
- 维护 fullscreen watch-progress reporter
- 把 raw progress sample 转成 replace-latest pending state
- 通过 shared telemetry queue flush

不负责：

- active video 身份判断
- 生成 `watchSessionId`
- 播放器状态管理
- fullscreen row selection
- 学习互动事件
- 业务完成状态计算

### 3.4 shared telemetry

`src/shared/telemetry` 只提供通用内存队列和 flush runtime：

- `createInMemoryTelemetryQueue(...)`
- `flushTelemetryQueue(...)`
- dedupe / upsert / merge
- retry backoff
- flush concurrency 和 batch size 控制

它不理解 watch-progress，也不 import 具体业务 endpoint。

## 4. 当前 watch-progress API 契约

后端语义 endpoint：

```text
POST /api/video-watch-progress
```

前端 request descriptor path：

```text
/video-watch-progress
```

请求体：

```ts
type WatchProgressRequestBody = {
  video_id: string;
  watch_session_id: string;
  position_ms: number;
  active_watch_ms: number;
  occurred_at: string;
  source_surface: 'detail' | 'feed' | 'fullscreen';
  client_context: {
    platform: string;
    app_version: string | null;
    os_version: string | null;
    device_model: string | null;
  };
};
```

成功响应：

```ts
type WatchProgressWriteResponse = {
  accepted: boolean;
};
```

字段语义：

| 字段 | 来源 | 语义 |
| --- | --- | --- |
| `video_id` | active row progress sample | 被观看的视频 ID，必须放在 body 中 |
| `watch_session_id` | active video visit 层 | 一次 active video visit 的关联 ID |
| `position_ms` | active row progress snapshot | 当前播放位置，单位 ms |
| `active_watch_ms` | watch-progress reporter 内存态累计 | 本次 active visit 内真实播放墙钟时长，单位 ms |
| `occurred_at` | reporter 当前时间 | 这条 pending state 最近一次被接受的时间 |
| `source_surface` | reporter 配置，fullscreen 默认 `fullscreen` | 上报来源表面 |
| `client_context` | `shared/lib/client-environment` mapper | 稳定客户端环境信息 |

不发送：

- `user_id`
- `duration_ms`
- `watch_ratio`
- `is_completed`
- `metadata.surface`
- route / screen / auth user / app session id
- locale / timezone / network / screen size

`durationSeconds` 仍会作为 raw sample 输入进入 reporter，但只用于前端 near-completion flush hint，不进入 request body。

## 5. 服务端完成状态

完成状态由后端计算，前端不再把 `is_completed` 放进 payload。

后端使用 upsert 后的单调字段判断完成：

```text
computed_completed =
  new_active_watch_ms > 10_000
  and duration_ms > 0
  and new_max_position_ms / duration_ms >= 0.9
```

这里的 `duration_ms` 来自 `catalog.videos.duration_ms`，不是前端请求。`new_active_watch_ms` 和 `new_max_position_ms` 都由后端 session upsert 保证只增不减。这样旧请求重试、seek 回退、乱序上报都不会让完成状态回落，也不会重复增加完成计数。

前端仍保留相同阈值的 near-completion 判断，但它只是 flush hint：

- 第一次达到 `active_watch_ms > 10_000 && positionMs / durationMs >= 0.9` 时立即写入 pending state 并 flush
- 后续仍处于 near-completion 区间的 samples 不重复 immediate flush
- 后续 samples 仍可按普通 throttle 更新 latest pending state

## 6. 数据流

当前 fullscreen watch-progress 的调用链：

```text
PlayableVideoSurface
  -> RowPlaybackMediaLayer
  -> FullscreenVideoRow
  -> FullscreenVideoPager
  -> FullscreenVideoSession
  -> useVideoWatchProgressReporter.reportSample(...)
  -> createWatchProgressTelemetryItem(...)
  -> shared telemetry queue upsert
  -> flushTelemetryQueue(...)
  -> reportVideoWatchProgress(body)
  -> reportMockVideoWatchProgress(body)
```

关键原则：

- player surface 只暴露播放事实，不知道 watch-progress API
- row 只负责 active row 内部装配
- pager 只做 active row gating 和 sample 转发
- session 持有 reporter、screen focused 定时 flush、screen blur / unfocus cleanup flush 和 active video switch flush
- reporter 负责 sample 过滤、累计、throttle、pending body 生成
- queue 负责 upsert、merge、retry、flush
- repository 当前负责 mock transport；真实后端接入时只替换 repository transport

## 7. pending state 设计

watch-progress 使用 replace-latest pending state，不使用 append-only 事件流。

队列 item payload：

```ts
type WatchProgressTelemetryPayload = {
  body: WatchProgressRequestBody;
};
```

dedupe key：

```text
video.watch_progress:${body.video_id}:${body.watch_session_id}
```

同一个 `video_id + watch_session_id` 只保留一条 pending state。新的 accepted sample 覆盖旧的 pending body。merge 不保留前端完成状态，因为完成状态由服务端计算。

这意味着 watch-progress 表达的是“本次观看 session 当前最新汇总状态”，不是完整行为日志。

## 8. `active_watch_ms` 计算

`active_watch_ms` 在 reporter 内按 `videoId + watchSessionId` 维护内存态：

```ts
type WatchProgressSessionStats = {
  activeWatchMs: number;
  lastSampleAtMs: number | null;
  lastPositionMs: number | null;
};
```

更新时机：

- 每个有效 raw sample 进入 reporter 后，先更新 session stats
- throttle 只控制是否写入 pending state，不阻止 stats 累计
- flush 不重新读取播放器，也不重新计算观看时长

累计规则：

- 第一个有效 raw sample 只建立 baseline，`active_watch_ms = 0`
- 后续 sample 只有在 position 正向推进且时间间隔可信时累计
- 累计值使用墙钟 elapsed time，不使用 position delta
- `playbackRate` 只用于判断 position 推进是否合理
- 非法 `playbackRate` 降级为 `1`
- pause / no movement 不累计
- seek forward 大跳变不累计
- seek backward 不累计
- 后台或长时间 gap 不累计
- 不按视频 duration 封顶，允许同一 active visit 内重复观看片段

当前可信间隔和漂移参数：

- `maxActiveWatchSampleGapMs = 2500`
- `playbackProgressDriftRatio = 1.5`

这些参数的目的不是精准识别所有播放器状态，而是在不接 AppState、不接播放状态事件的 MVP 中避免明显误算。

## 9. sample 接受与 flush 时机

raw sample 可以高频进入 reporter，例如播放器 `timeUpdate`。

pending state 写入节流：

- 普通 progress sample：同一 `videoId + watchSessionId` 最多每 `1s` 写入一次 pending state
- near-completion flush hint：绕过 `1s` throttle，立即写入 pending state

flush 时机：

- video screen focused 时每 `10s` 定时 flush
- active video switch 前 flush 已有 pending state
- 首次 near-completion flush hint 后立即 flush
- video screen blur / unfocus 时 flush 已有 pending state，并同时暂停播放、停止采样转发、停止定时 flush

不触发 flush：

- 用户 pause
- 用户 resume
- seek
- 字幕点击
- 弹窗打开/关闭

flush 语义：

- flush 只发送已经存在的 pending state
- flush 不更新 stats
- flush 不重新读 player
- flush 不保证 App 被杀前一定成功

## 10. 失败与重试

watch-progress 走 shared telemetry queue。

当前行为：

- sender 失败不会 toast
- retryable 失败保留在内存队列中等待后续 flush
- flush 有并发和单批上限
- App 被杀后内存队列丢失

repository 当前仍是 mock-backed：`reportVideoWatchProgress(body)` 调用 `reportMockVideoWatchProgress(body)` 并返回 `{ accepted: true }`。它不读取 `EXPO_PUBLIC_API_BASE_URL`，不要求 auth token，不验证 UUID 或 catalog video existence。

`createWatchProgressRequest(body)` 仍保留 `/video-watch-progress` descriptor，用来锁定未来真实 API 的请求形状。后续接真实后端时，只替换 repository transport，不改 reporter、pager 或 payload shape。

## 11. 未来学习事件上报预留

未来学习互动事件包括但不限于：

- 单词 lookup
- 字幕 token 点击
- sentence replay
- exposure
- self mark mastered
- quiz answer
- save / collect learning item

这些事件不应塞进 watch-progress body。推荐新增独立 feature：

```text
src/features/learning-interactions/
  api/
  model/
  README.md
```

学习事件可以复用 active video visit 层的 `watchSessionId`：

```ts
type LearningInteractionBase = {
  video_id: string;
  watch_session_id: string;
  occurred_at: string;
  source_surface: 'fullscreen' | 'detail' | 'feed';
  client_context: AnalyticsClientContext;
};
```

但它应该拥有独立事件语义和独立 pending 结构。

## 12. watch-progress 与 learning-interactions 的差异

| 维度 | watch-progress | learning-interactions |
| --- | --- | --- |
| 当前状态 | 已落地 fullscreen MVP | 未实现 |
| 数据模型 | replace-latest session summary | append-only 或按事件类型聚合 |
| dedupe key | `video_id + watch_session_id` | 由事件类型决定 |
| flush 触发 | focused 10s、切换、near-completion、screen blur / unfocus | 可复用 telemetry flush，但由 feature 自己决定 |
| completed 语义 | 服务端计算 | 不应判断观看完成 |
| active_watch_ms | 维护并上报 | 可引用但不维护 |
| watchSessionId | 消费 active visit ID | 复用 active visit ID |
| API body | 固定 watch-progress contract | 独立 learning interaction contract |

学习事件的典型选择：

- lookup / token click：append-only，保留每次交互
- exposure：可按窗口聚合或采样，避免高频刷屏
- self mark mastered：interactive-write 或 telemetry 需要单独判断；如果用户需要立即看到稳定状态，应走 interactive-write
- quiz answer：通常是 interactive-write 或学习结果事件，不应混进 watch-progress

## 13. 推荐新增 learning-interactions 的落地方式

当开始实现学习事件时，按以下顺序推进：

1. 在 active row 或 session 层确认事件来源能拿到 `watchSessionId`
2. 新建 `features/learning-interactions`
3. 定义事件类型和最小 request body
4. 复用 `client-environment` mapper 生成 `client_context`
5. 复用 `shared/telemetry` queue，但不要复用 watch-progress queue item 类型
6. 为不同事件选择 append-only 或 aggregated pending state
7. 在 docs 中补充每类事件的 dedupe / merge / flush 语义

不建议一开始做统一大而全的 `events` endpoint。MVP 阶段应先让 watch-progress 和 learning-interactions 在 feature 边界上分开，只共享底层 telemetry 能力和 active visit correlation。

## 14. 测试要求

watch-progress 已覆盖的关键测试方向：

- request body 使用 `video_id` 和 `active_watch_ms`
- request body 不发送 `duration_ms` 和 `is_completed`
- repository mock-backed，并保留 `/video-watch-progress` request descriptor
- invalid sample 被忽略
- raw sample 在 throttle 前更新 active watch stats
- pause、seek、background gap 不累计 active watch
- 2x playback 按墙钟观看时长累计
- near-completion flush hint 只触发一次 immediate flush
- telemetry merge 保留 latest body，不 sticky 前端完成状态
- pager / row 只转发 `videoId`、`watchSessionId`、`playbackRate` 和 progress sample，不计算 watch-progress state

未来 learning-interactions 至少需要覆盖：

- 无 `watchSessionId` 时是否丢弃或降级
- 每类事件的 dedupe / merge 规则
- append-only 事件不会被错误覆盖
- aggregated 事件不会无限增长
- 与 watch-progress 队列互不影响
- flush 失败不打断用户交互

## 15. 当前维护原则

- watch-progress 是 session summary，不是完整播放器日志
- `watchSessionId` 是 active video visit 关联 ID，不属于 watch-progress feature
- `active_watch_ms` 由 watch-progress reporter 维护，不由 pager 维护
- `client_context` 来自通用 client-environment，不由业务 caller 临时拼
- completed 状态由后端计算，前端 near-completion 只影响 flush 时机
- future learning events 只共享 correlation，不共享 watch-progress payload
- 新增事件类型时先定义 feature 边界，再定义 flush / merge 语义
