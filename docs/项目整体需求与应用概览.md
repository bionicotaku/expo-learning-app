# 产品总览

## 1. 文档定位

本文档用于定义项目的产品总览，回答四个问题：

- 这个项目到底要做什么
- 这个 App 在用户视角下是什么样的产品
- MVP 需要先闭环哪些能力
- 产品应该按什么阶段逐步扩展

本文档只负责产品定义、范围边界和阶段目标，不承担技术实现细节。技术实现与依赖选择单独见 [技术栈选型](./技术栈选型.md)。

## 2. 项目一句话定义

这是一个面向中文学习者或英语学习者的内容驱动型学习 App：用户通过连续刷看的短视频内容进入学习场景，在播放过程中理解字幕、提取知识点，再把有价值的内容沉淀到个人学习空间中。

它不是单纯的播放器，也不是单纯的背单词工具，而是一个以内容消费为入口、以学习沉淀为结果的移动端学习产品。

## 3. 产品目标

### 3.1 核心目标

- 让用户以接近内容平台的低门槛方式进入学习
- 让“看内容”与“学内容”发生在同一个连续体验里
- 让学习结果能够被保存、回看、整理和复习
- 让应用从一开始就具备真实账户体系，而不是匿名演示 Demo

### 3.2 产品成功标准

当前阶段的产品成功，不以“功能很多”为标准，而以“主路径闭环”作为标准。MVP 至少需要形成这条链路：

1. 用户完成登录并恢复会话
2. 进入内容流并连续浏览视频
3. 在视频场景中完成基础理解和播放控制
4. 把值得保留的学习内容加入个人沉淀空间
5. 在个人沉淀区回看这些内容

如果这条链路没有闭环，那么产品仍然只是原型，而不是可持续使用的学习应用。

## 4. 目标用户与核心场景

### 4.1 目标用户

当前产品主要面向以下用户：

- 希望通过短内容而不是长课程进入学习状态的用户
- 愿意利用碎片时间学习英语或其他语言内容的用户
- 不满足于“只看视频”，希望把内容中的词汇、短语、表达或知识点沉淀下来的人
- 希望在移动端长期积累学习结果的用户

### 4.2 核心使用场景

典型场景包括：

- 通勤、排队、休息等碎片时间，快速进入内容流学习
- 在连续刷看中遇到值得学习的表达并即时保存
- 回到个人沉淀区查看已保存内容
- 通过历史、收藏、复习列表重新进入内容

## 5. 产品价值主张

### 5.1 对用户的直接价值

- 降低开始学习的心理门槛
- 让学习行为更接近日常内容消费习惯
- 把一次性观看转化成可积累的学习资产
- 让个人学习空间成为长期复用的知识入口

### 5.2 与传统学习工具的差异

和传统“从词汇表或练习题开始”的学习工具相比，这个产品的入口不是题库，而是内容流。学习辅助和学习沉淀并不是独立页面的附属能力，而是围绕内容消费主线展开。

## 6. App 形态与信息结构

### 6.1 用户视角下的 App

从用户视角看，App 的整体形态可以描述为：

- 有真实登录能力的移动端学习应用
- 首屏是可连续浏览的内容流
- 观看过程中逐步叠加学习辅助层
- 学习结果进入个人沉淀区
- 用户在个人空间管理会话、偏好和回看入口

### 6.2 主应用结构

当前项目的目标结构分为三大业务区：

### 认证区

负责用户进入产品前后的身份状态管理，包括：

- 邮箱密码登录
- 会话恢复
- 退出登录
- 忘记密码和重置密码
- 受保护页面的访问分流

认证不是孤立页面，而是决定主应用区是否可访问的入口控制层。

### 主内容区

主内容区是产品第一优先级，也是学习主线所在区域。

它承载：

- 连续视频流
- 视频播放与切换
- 详情或沉浸式观看入口
- 基础学习辅助能力
- 回到列表后的状态保持

在大多数时间里，用户对产品的核心感知都发生在这里。

### 学习沉淀区

学习沉淀区负责承接用户在内容流里产生的有效学习结果，包括：

- 已保存条目列表
- 条目详情
- 收藏、历史、复习入口
- 学习进度与回看能力

这一部分决定了产品是否能从“内容体验”成长为“长期学习工具”。

### 个人空间

个人空间负责承载和用户本人相关的系统级内容，包括：

- 当前账户信息
- 会话与退出登录
- 偏好设置
- 历史入口
- 收藏入口
- 反馈、帮助、关于等扩展入口

## 7. 核心业务闭环

### 7.1 主路径

当前项目要形成的核心业务闭环是：

1. 用户打开 App
2. 系统恢复会话或进入登录流程
3. 用户进入内容流
4. 用户观看视频并完成基础理解
5. 用户把有价值的学习内容沉淀到个人空间
6. 用户在沉淀区查看和管理已保存内容
7. 用户通过历史或收藏再次回到内容

### 7.2 为什么这条闭环重要

如果只有内容流而没有沉淀能力，产品会停留在“刷视频”。
如果只有沉淀区而没有内容流，产品会失去进入学习的自然入口。
如果没有登录和会话恢复，沉淀区就无法成为长期资产。

所以对这个项目来说，内容流、认证和沉淀区不是并列的三个可选功能，而是一个必须闭环的最小系统。

## 8. MVP 定义

### 8.1 MVP 必须完成的范围

第一阶段 MVP 应至少完成以下能力：

### 认证闭环

- 邮箱密码登录
- 会话恢复
- 退出登录
- 认证区与主应用区的受保护跳转

### 内容流闭环

- 连续竖滑视频流
- 真实 feed 读取与刷新行为
- 当前页自动播放
- 默认静音与点击切换静音
- 基础加载态与错误态

### 内容理解的最小辅助层

- 当前视频的标题与补充说明
- 后续可承接字幕、词汇解释和知识点抽取的结构位置

MVP 阶段不要求一次性把所有学习辅助能力做满，但必须预留后续扩展边界。

### 学习沉淀的最小闭环

- 用户可以保存学习条目
- 用户可以查看已保存条目列表
- 用户可以再次进入条目或关联内容

### 个人空间的最小闭环

- 查看当前账户基础信息
- 退出登录
- 展示收藏、历史、设置等入口骨架

### 8.2 MVP 非目标

以下内容不属于当前 MVP 必须完成项：

- 推荐算法或个性化排序
- 复杂社交互动
- 完整离线模式
- 复杂内容生产后台
- 多端同步冲突处理
- Web 端正式支持

当前 MVP 的主要目标仍然是移动端主链路闭环，而不是把所有二期能力提前做完。

### 8.3 MVP 数据策略与实现要求

MVP 阶段的数据策略需要明确区分“数据来源是 mock”与“功能只是演示”这两件事。

当前约束是：

- 除 `Supabase Auth` 外，MVP 阶段的业务数据全部采用 mock 数据
- 但所有核心业务流必须按真实产品方式实现，不能只做静态页面展示

这意味着：

- Feed 要有真实的读取、加载、错误和刷新语义，而不是只把几条内容写死在页面里
- 详情、收藏、历史、沉淀区等业务流要有真实的读取与写入行为，即使底层暂时写入的是 mock 层
- 页面必须具备 loading、empty、error、success、mutation in-flight 这类运行态，而不是只有设计稿态
- 用户操作要真正改变应用状态，例如保存条目、取消保存、进入详情、回到列表、查看历史

换句话说，MVP 可以不接真实内容服务，但不能做成“只有展示没有行为”的前端壳。

### 8.4 为什么这样要求

这样做的目的有两个：

- 第一，尽早验证产品主链路和状态流转本身是否成立
- 第二，后续接入真实服务时，替换的是数据来源，而不是推翻页面结构和业务交互

如果 MVP 只是静态展示，那么后续接真实接口时，几乎所有页面、状态管理和交互逻辑都要重写，前期投入价值会很低。

## 9. 分阶段建设建议

### 9.1 第一阶段：账户与主应用壳

目标是先搭出完整 App 壳，让产品具备真实的入口控制和应用结构。

这一阶段应优先完成：

- 认证区
- 会话恢复
- 受保护路由
- 主应用 Tabs 骨架

### 9.2 第二阶段：学习闭环最小实现

目标是把内容消费变成最小学习闭环。

这一阶段应优先完成：

- 保存学习条目
- 沉淀区列表
- 条目详情
- 历史与收藏入口

### 9.3 第三阶段：学习辅助增强

目标是让产品从“可保存”升级成“可理解、可复习”的学习工具。

这一阶段可逐步补强：

- 字幕层
- 词汇与表达解释
- 视频详情页
- 沉浸式播放能力

### 9.4 第四阶段：数据与体验增强

目标是进一步提升产品完成度和长期使用价值。

包括但不限于：

- 真实内容服务接入
- 学习记录与统计
- 更完整的个性化能力
- 更细的缓存与本地数据策略

## 10. 运行范围与平台边界

当前项目的优先平台是：

- iOS
- Android

当前阶段不把 Web 作为正式产品目标。虽然 Expo 具备 Web 能力，但该能力不属于当前产品成功标准。

## 11. 文档使用建议

新接手这个项目时，建议按下面顺序阅读：

1. 先读本文档，理解产品目标、范围和阶段边界
2. 再读 [技术栈选型](./技术栈选型.md)，理解为什么采用当前实现方向
3. 再读 [项目规范](./项目规范.md)，理解后续实现默认应遵循的结构与测试标准
4. 最后再进入代码与模块设计

## 12. 总结

当前项目的目标不是做一个“能播放视频的 Expo Demo”，而是构建一个以短视频内容为入口、以学习沉淀为结果的移动端学习产品。

MVP 的关键不在于一次性做很多功能，而在于尽快闭合这条主线：

- 登录进入
- 内容消费
- 学习辅助
- 学习沉淀
- 个人回看

只要这条闭环成立，后续的字幕、个性化、历史、复习和数据能力都可以在当前结构上继续演进。
