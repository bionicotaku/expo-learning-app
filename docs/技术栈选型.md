# 技术栈选型

## 1. 文档目标

本文档定义当前项目的推荐技术栈，并明确以下问题：

- 哪些技术方向已经确定
- 为什么这样选
- 每一层技术分别承担什么职责
- 哪些方案暂时不选
- 这一套选择如何支撑 MVP 和后续演进

本文档服务于当前项目，而不是通用模板。产品目标与范围定义见 [产品总览](./项目整体需求与应用概览.md)，具体目录与实现约束见 [项目规范](./项目规范.md)，网络层与事件通道的处理方式见 [网络层与事件上报规范](./网络层与事件上报规范.md)。

## 2. 选型原则

当前项目的技术选型遵循以下原则：

1. 以 iOS / Android 移动端为主，不为了 Web 额外复杂化当前架构
2. 优先采用 Expo 当前官方推荐路径，而不是回退到手工维护更多底层细节
3. MVP 只接入 `Supabase Auth`，但把耦合集中在认证层，避免散落到业务页面
4. 不为了“以后可能会换”而过度抽象，也不把第三方 SDK 直接塞满全项目
5. 优先保证演进成本低，而不是一开始做成大而全架构
6. 区分“当前必须落地的标准”和“未来可引入的能力”，不把候选项写成既定事实

## 3. 推荐技术栈总览

当前推荐组合如下：

- 客户端基础：Expo + React Native + TypeScript
- 导航：Expo Router
- 当前最小运行态导航结构：Root Stack + Auth entry route + NativeTabs app shell + Fullscreen Video detail route
- 后续完整扩展形态：Protected Routes + Tabs + tab 内 Stack
- 认证：Supabase Auth
- 认证客户端：`@supabase/supabase-js`
- 认证相关配套：`react-native-url-polyfill/auto` + `expo-secure-store`
- 服务端数据状态：`@tanstack/react-query`
- 本地交互状态：React hooks + `useReducer` + 少量 Context
- 媒体播放：`expo-video`
- 本地数据层：后续按需引入 `expo-sqlite`
- 表单与校验：后续按需引入 `react-hook-form` + `zod`
- 工程组织：轻量 FSD

需要特别补充一条实现约束：

- MVP 阶段除 `Supabase Auth` 外，业务数据全部使用 mock 数据源，但接口与功能按真实异步业务流实现，不做纯展示页

## 4. 客户端基础层

### 4.1 选择

- Expo
- React Native
- TypeScript

### 4.2 选择原因

当前项目是移动端内容学习应用，核心需求集中在：

- iOS / Android 双端交付
- 媒体播放
- 路由与导航
- 原生模块能力
- 快速验证与快速迭代

Expo 当前仍推荐从 `create-expo-app` 默认模板起步；默认模板已经集成 Expo Router。官方文档也明确把 Expo 定位为“让 Android 和 iOS 开发更容易的 React Native 框架”。

参考：

- [Expo Create a project](https://docs.expo.dev/get-started/create-a-project/)
- [Expo Router introduction](https://docs.expo.dev/router/introduction/)

### 4.3 不选什么

当前阶段不建议切换到：

- 裸 React Native CLI
- Flutter
- 手工维护完整 React Navigation 树的非 Router 方案

原因不是这些技术不能做，而是它们不能给项目带来足够的实际收益，反而会打断 Expo 的迭代效率和默认能力。

## 5. 导航与路由

### 5.1 选择

- Expo Router
- 根布局使用 `Stack`
- 主应用区先使用 `NativeTabs`
- 当前最小运行态先保持：
  - `Auth`
  - `Feed / Save / Me` 三个 tabs
  - `Fullscreen Video`
- 后续引入认证区与多一级主导航时，再补：
  - `Protected Routes`
  - `Tabs`
  - tab 内 `Stack`

### 5.2 当前最小运行态结构

当前已对齐的最小运行态路由结构如下：

```text
src/app/
  _layout.tsx
  index.tsx
  auth.tsx
  (tabs)/
    _layout.tsx
    feed.tsx
    save.tsx
    me.tsx
  video/
    [videoId].tsx
```

### 5.3 后续完整扩展形态

如果后续重新引入认证与多一级主导航，再扩展为：

```text
src/app/
  _layout.tsx
  (auth)/
    _layout.tsx
    sign-in.tsx
    forgot-password.tsx
  (tabs)/
    _layout.tsx
    feed/
      _layout.tsx
      index.tsx
    collections/
      _layout.tsx
      index.tsx
    profile/
      _layout.tsx
      index.tsx
  video/
    [videoId].tsx
```

### 5.4 选择原因

Expo Router 是当前 Expo 官方主推的文件路由方案，并且构建在 React Navigation 之上。它既保留 React Navigation 的能力，又减少了手工维护路由树、类型和深链映射的成本。

对于“先通过登录页进入主应用 tabs，再从 feed 进入内容详情页”的组合，这条路线最稳。

`Protected Routes` 是 Expo Router 当前官方提供的认证分流方式，适合直接表达“已登录才能进主应用”“未登录只能进认证区”这类规则。

参考：

- [Expo Router SDK](https://docs.expo.dev/versions/latest/sdk/router/)
- [Protected Routes](https://docs.expo.dev/router/advanced/protected/)

### 5.5 当前落地要求

- `app/` 目录只放 route 与 `_layout`
- 业务组件、hooks、实体类型、feature 逻辑不放在 `app/`
- 页面结构由 route group 表达，业务实现继续放在 `src/pages`、`src/features`、`src/widgets` 等目录

## 6. 认证与会话

### 6.1 MVP 选择

- Supabase Auth
- `@supabase/supabase-js`

MVP 只接一个外部身份系统，就是 Supabase Auth。

### 6.2 为什么选 Supabase Auth

Expo 官方在认证指南中明确提到：如果想更快完成邮箱密码认证，可以直接使用带内建认证能力的服务，Supabase 是被直接列出的选项之一。

对当前 MVP 来说，这比自建认证系统或同时对接多种身份服务更合理。

参考：

- [Authentication in Expo and React Native apps](https://docs.expo.dev/develop/authentication/)

### 6.3 耦合控制方式

当前不做“为了以后可能替换”而设计的重型接口层，但也不把 Supabase 客户端直接散落到页面里。

推荐边界如下：

```text
src/shared/lib/supabase/
  client.ts

src/features/auth/api/
  sign-in.ts
  sign-out.ts
  reset-password.ts

src/features/auth/model/
  session-provider.tsx
  use-auth-session.ts
```

约束如下：

- 只有 `shared/lib/supabase` 负责创建 client
- 只有 `features/auth` 直接调用 `supabase.auth.*`
- 页面、路由保护和其他 feature 只消费认证状态与认证动作

这样做的结果是：

- 当前开发不会被抽象层拖慢
- 未来如果替换认证供应商，重构面基本集中在认证层
- 业务页面不直接绑定 Supabase SDK 细节

### 6.4 会话持久化与环境变量

当前推荐：

- Native 端会话持久化使用 `expo-secure-store`
- 引入 `react-native-url-polyfill/auto`
- 环境变量使用 `EXPO_PUBLIC_SUPABASE_URL`
- 优先使用 `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY`

Supabase 新的 Expo 快速开始文档已经转向 `publishable key` 命名，而不是继续把旧的 `anon key` 当成唯一标准。
如果后续引入邮件回调、密码重置链接或 OAuth，再补深链 scheme 与跳转配置。

参考：

- [Use Supabase with Expo React Native](https://supabase.com/docs/guides/getting-started/quickstarts/expo-react-native)
- [Build a Social Auth App with Expo React Native](https://supabase.com/docs/guides/auth/quickstarts/with-expo-react-native-social-auth)
- [Native Mobile Deep Linking](https://supabase.com/docs/guides/auth/native-mobile-deep-linking)
- [Expo SecureStore](https://docs.expo.dev/versions/latest/sdk/securestore/)

### 6.5 为什么当前不用更重的抽象

当前阶段不建议引入：

- `AuthRepository` / `AuthGateway` / 多供应商适配层
- 大量 provider interface
- 过早的 domain service 层

因为当前业务真相很简单：MVP 只接 Supabase Auth。
正确的做法是把第三方耦合收口，而不是为了“将来可能换”提前堆一层又一层抽象。

## 7. 服务端数据状态与本地状态

### 7.1 选择

- `@tanstack/react-query` 负责 source/server state cache
- `zustand` 负责 source-agnostic 的 `video-runtime`
- React Context + hooks + `useReducer` 继续负责认证和局部复杂交互状态

### 7.2 为什么这样分

远程数据和本地交互状态本质上不是一类问题。

对于这个项目：

- Feed 数据、条目列表、历史、收藏同步等属于服务数据
- `videoId -> runtime override` 属于 source-agnostic 客户端状态
- 播放状态、当前页面状态、表单输入、局部交互属于其余本地状态

把这两类状态混在同一种 store 里，后续会越来越难维护。
所以更合理的边界是：

- React Query 管“服务数据”
- Zustand 管 `video-runtime` 这类 keyed runtime state
- hooks / reducer / Context 管其余本地交互

TanStack Query 官方也单独提供了 React Native 场景下的在线状态与 AppState 对接方式，说明这条路径是成熟的。

参考：

- [TanStack Query React Native](https://tanstack.com/query/v5/docs/framework/react/react-native)

### 7.3 MVP 的 mock 数据实现方式

MVP 虽然不接真实业务后端，但不意味着可以直接把数据常量写死在页面里。

推荐做法是：

- 由 feature 或 entity 层提供明确的 mock API / mock repository
- 所有读取与写入都通过异步函数返回 `Promise`
- 保留分页、加载、错误、空状态、写操作进行中等运行态
- 页面只消费 query / mutation 或 feature action，不直接内嵌 mock 常量

例如：

- feed 列表通过 mock page fetch 返回
- fullscreen 的 `like / favorite` 当前先走本地 runtime toggle
- 沉淀区列表通过 mock query 拉取
- 历史记录通过 mock read / write 更新

这样做的目的不是为了模拟一切后端细节，而是为了让页面、状态层和交互链路从一开始就按真实产品方式组织。

### 7.4 为什么 MVP 仍然值得引入 React Query

即使在 mock 数据阶段，React Query 仍然是合适的，因为当前要验证的不是“接口是否已经上线”，而是：

- 列表如何加载和分页
- 详情如何拉取
- 写操作如何提交和回写
- 错误态和重试如何处理
- 未来切换真实服务时，页面是否可以不重写

如果 MVP 只做同步常量展示，那么后续接真实接口时，状态模型和页面交互往往要整块返工。
因此，当前推荐的做法是“先用 mock 数据，把真实的异步交互链路做完整”。

### 7.5 当前对 Zustand 的正式定位

当前项目不把 Zustand 视为“所有状态的总仓库”。

但对未来已经明确的一类状态，方案现在已经锁定：

- source-agnostic
- `videoId` 维度 keyed map
- 跨页面复用
- 纯客户端 runtime override

这类状态固定采用 Zustand。

也就是说，未来的职责分工固定为：

- 服务端来源状态：React Query
- `video-runtime` 这类客户端 runtime state：Zustand
- 局部页面交互：hook / reducer
- 认证与少量应用级状态：Context

这里的关键不是“项目全面改用 Zustand”，而是：

- 不再让 React Query 同时承担 source state 和 runtime override
- 也不再把 runtime override 混回 entity 或单个 source feature

## 8. 媒体层

### 8.1 选择

- `expo-video`

### 8.2 选择原因

当前项目的核心能力是视频内容流。Expo 当前官方已将视频能力收敛到 `expo-video`，不应再以旧的 `expo-av` 作为新项目的默认视频方案。

参考：

- [Expo Video](https://docs.expo.dev/versions/latest/sdk/video/)

### 8.3 当前媒体层定位

`expo-video` 在本项目里不是“页面里的一个小组件”，而是媒体基础设施的一部分。
它需要和以下能力一起被考虑：

- 内容流分页
- 当前页播放策略
- 静音与播放控制
- 后续详情页或沉浸式页扩展

### 8.4 当前阶段不提前做什么

以下媒体能力先不作为 MVP 前置条件：

- 后台播放
- 画中画
- 横竖屏联动
- 大规模播放器池复用

这些能力未来可以做，但不应阻塞当前的主链路闭环。

## 9. 本地数据层

### 9.1 当前策略

当前 MVP 不预先锁定 ORM。

本地数据层的推荐顺序是：

1. 先把认证、主路由、内容流和沉淀闭环跑通
2. 明确哪些数据必须本地持久化
3. 再引入 `expo-sqlite`
4. schema 真的复杂后，再决定是否增加 ORM / migration 封装

### 9.2 为什么后置

当前项目还没有进入“复杂本地数据驱动”的阶段。
如果现在就把 ORM、仓储层、迁移工具一起压进来，绝大多数只会成为提前设计。

### 9.3 适合本地化的数据

后续可能进入本地数据库的内容包括：

- 历史记录缓存
- 收藏和沉淀条目镜像
- 字幕缓存
- 学习进度
- 离线可读数据

## 10. 表单与验证

当前建议是：

- 等认证页和设置页真正落地时，再引入 `react-hook-form`
- 同时使用 `zod` 做输入校验

这两个库当前不是前置依赖，但属于明确的后续优选组合。

## 11. 测试与验证

### 11.1 当前组合

当前推荐的验证组合是：

- 纯逻辑测试：Vitest
- 类型检查：`npx tsc --noEmit`
- 代码规范：`npm run lint`
- 原生与路由验证：`npm run ios`

### 11.2 推荐策略

短期内继续保持：

- 纯逻辑单元测试覆盖 reducer、helper、mock API 和业务规则
- 类型检查作为默认快速回归的一部分
- 页面、路由或原生播放改动后补做 iOS 启动验证

如果后续需要更多页面与认证流交互测试，再按需补充 React Native 组件级测试方案。当前阶段不必为了“完整测试栈”提前引入更多重量。

### 11.3 当前回归命令

项目默认回归命令定义为：

- `npm run quick-check`
- `npm run check`

这两条命令是后续开发的默认校验入口。

## 12. 工程组织方式

### 12.1 选择

当前项目继续采用轻量 FSD：

- `app`
- `pages`
- `widgets`
- `features`
- `entities`
- `shared`

### 12.2 为什么继续用这套结构

当前业务具有以下特点：

- 页面不算少，但每个页面内会混合播放、认证、分页、沉淀等多种职责
- 后续模块会逐步增加
- 路由层和业务层必须严格分开

轻量 FSD 可以让这些职责保持可读边界，同时不会重到需要额外维护复杂架构层级。

### 12.3 当前目录约束

- `src/app` 只放 route 和 layout
- `src/pages` 负责页面组合
- `src/widgets` 负责页面级复合视图
- `src/features` 负责用户交互能力
- `src/entities` 负责领域实体与类型
- `src/shared` 负责通用基础设施

## 13. 当前明确不纳入标准栈的内容

当前不纳入默认技术标准的内容包括：

- 任何预设 UI 组件库
- 预先设计的大型仓储接口层
- 预先引入 ORM 与复杂迁移体系
- 非必要的自定义原生模块
- 为了 Web 兼容而增加的当前不必要复杂度

## 14. 最终结论

对于当前项目，最合适的组合不是“找一套最炫的新栈”，而是采用一套清晰、克制、可演进的产品化收敛：

- 用 Expo Router 承担导航与认证分流
- 用 Supabase Auth 完成 MVP 认证闭环
- 用 React Query 承担服务数据层
- 用 Zustand 承担 source-agnostic 的 video runtime state
- 用 hooks / reducer / Context 管理其余本地交互状态
- 用 expo-video 承担媒体基础设施
- 用轻量 FSD 保持代码边界清楚

这套组合的优点是：

- 符合 Expo 当前官方推荐方向
- MVP 可以快速落地
- 后续重构成本可控

对这个项目来说，这比“先设计一个未来十倍复杂度的架构”更正确。
