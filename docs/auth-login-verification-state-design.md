# 登录、验证与状态维护设计文档

## 1. 文档目标

本文档描述一个可复用的认证系统设计，适用于移动端应用、Web 应用或前后端分离产品。重点不是具体技术实现，而是从产品需求、业务流程、状态模型和系统边界出发，定义一套完整的登录、验证和认证状态维护方案，供其他项目直接参考或裁剪。

本文档覆盖以下场景：

- 邮箱 + 密码登录
- 邮箱验证码登录
- 忘记密码与密码重置
- 新用户首次验证后的密码设置
- 会话恢复、会话失效和退出登录
- 认证状态同步、风控冷却和异常处理

## 2. 设计目标

### 2.1 业务目标

- 让用户可以通过低门槛方式完成首次进入产品
- 让老用户可以快速、安全地恢复访问
- 在“已验证但尚未完成账户初始化”的中间状态下，仍能控制访问范围
- 确保认证状态在页面刷新、应用重启、多端登录、网络波动后仍保持一致

### 2.2 用户体验目标

- 登录路径少，用户决策成本低
- 每一步状态明确，不让用户困惑于“我到底算不算登录成功”
- 用户输入错误、验证码错误、请求过快、网络失败时，都有明确反馈
- 登录成功后的跳转逻辑稳定，不出现重复跳转、跳错页面或闪屏

### 2.3 安全目标

- 防止暴力尝试密码和验证码
- 防止验证码被无限重发
- 防止仅凭本地 UI 状态进入受保护页面
- 确保退出登录后本地认证状态及时清空

## 3. 适用范围与非目标

### 3.1 适用范围

- 用户体系以邮箱身份为主
- 产品需要同时支持密码登录和验证码登录
- 产品允许“用户已通过身份验证，但还未完成密码设置”
- 客户端需要自行维护认证状态和页面准入控制

### 3.2 非目标

- 不限定具体认证供应商
- 不限定令牌格式、数据库结构和 SDK 选型
- 不覆盖企业级单点登录、组织权限、RBAC 细粒度授权等更上层能力

## 4. 核心业务概念

为了让不同项目可以共用这套设计，建议先统一以下概念。

### 4.1 身份验证成功

指系统确认“当前操作人确实拥有该邮箱的控制权”。

例如：

- 密码校验通过
- 验证码校验通过

### 4.2 账户初始化完成

指用户不仅完成了身份验证，还满足进入主应用的最低账户条件。

典型条件包括：

- 已设置登录密码
- 已接受必要协议
- 已完成基础资料

本文档以“是否已设置密码”作为最小完成条件。

### 4.3 会话有效

指客户端当前持有可用的认证凭证，并且服务端仍承认该凭证有效。

### 4.4 认证状态

建议把认证状态拆成两个层面，而不是用单一布尔值表示：

- 是否已通过身份验证
- 是否已完成账户初始化

这样可以避免“验证码登录成功后立即被当作完整登录用户”的状态混乱。

## 5. 状态模型设计

一个可复用的认证系统，至少应包含以下状态。

### 5.1 核心状态字段

- `isAuthInitialized`
  表示客户端是否已经完成首次认证状态探测
- `isAuthenticated`
  表示当前是否拥有有效身份
- `isAccountReady`
  表示当前账户是否满足进入主应用的条件
- `session`
  当前会话对象或会话摘要
- `user`
  当前用户信息摘要
- `accessToken`
  当前访问令牌
- `refreshToken`
  当前刷新令牌
- `expiresAt`
  会话过期时间

### 5.2 建议的状态判断规则

- 未初始化：`isAuthInitialized = false`
- 未登录：`isAuthInitialized = true && isAuthenticated = false`
- 已验证未完成：`isAuthenticated = true && isAccountReady = false`
- 已完整登录：`isAuthenticated = true && isAccountReady = true`

### 5.3 页面准入规则

建议不要只看“是否有 session”，而应看“身份状态 + 账户完成状态”。

- 公开页面：任何状态均可访问
- 认证流程页面：未登录或已验证未完成状态可访问
- 主应用页面：只有已完整登录状态可访问

## 6. 用户旅程设计

## 6.1 路径一：邮箱 + 密码登录

### 用户目标

老用户通过已设置的邮箱和密码直接进入产品。

### 基本流程

1. 用户输入邮箱和密码
2. 客户端进行基础格式校验
3. 服务端校验邮箱与密码
4. 校验成功后签发会话
5. 客户端更新认证状态
6. 若账户已完成初始化，则进入主应用

### 关键要求

- 密码错误时不能泄露账户存在性
- 登录成功后应立即建立统一会话
- 跳转主应用前，必须先完成本地认证状态更新

## 6.2 路径二：邮箱验证码登录

### 用户目标

用户不记得密码、首次体验产品，或更偏好轻量登录方式时，可以通过验证码完成身份验证。

### 基本流程

1. 用户输入邮箱
2. 系统发送验证码
3. 用户输入验证码
4. 服务端校验验证码
5. 校验成功后建立会话
6. 系统检查账户是否已完成初始化
7. 若已完成，则进入主应用
8. 若未完成，则进入账户初始化流程

### 关键要求

- 发送验证码前必须校验邮箱格式
- 验证码有效期、尝试次数和重发频率必须受控
- 验证码登录成功不等于一定可以直接进入主应用

## 6.3 路径三：忘记密码

### 用户目标

用户忘记原密码，但仍可证明自己拥有该邮箱。

### 基本流程

1. 用户进入忘记密码入口
2. 输入邮箱
3. 系统发送重置验证码或重置链接
4. 用户完成身份验证
5. 系统允许用户设置新密码
6. 新密码生效后建立或刷新会话
7. 用户进入主应用

### 关键要求

- 忘记密码是“身份验证后允许修改凭证”的流程，不应直接暴露账户细节
- 新密码设置成功后，应同步更新账户初始化状态
- 建议重置成功后使旧会话失效，至少在高安全等级产品中如此

## 6.4 路径四：首次验证后设置密码

### 用户目标

新用户通过验证码完成身份确认后，继续设置一个长期可用的密码，完成账户初始化。

### 基本流程

1. 用户通过验证码完成身份验证
2. 系统判定账户尚未完成初始化
3. 引导用户进入“设置密码”页面
4. 用户输入并确认新密码
5. 系统校验密码强度并保存
6. 更新账户完成状态
7. 用户进入主应用

### 关键要求

- 设置密码页面只能在已验证状态下访问
- 账户初始化完成前，不允许进入完整业务区
- 设置密码成功后，客户端和服务端状态都必须同步完成

## 7. 认证状态维护设计

## 7.1 单一真相来源

认证系统必须有唯一可信的状态来源。

推荐原则：

- 服务端负责“认证是否成立”
- 客户端负责“认证状态的缓存、派生和消费”
- 页面不应自己发明“我觉得用户已经登录”的局部状态

### 设计要求

- 所有登录、验证、登出行为都应最终汇总到统一认证状态容器
- 页面跳转应基于统一认证状态，而不是局部组件变量
- 认证 SDK 或会话系统的变化事件，应自动驱动客户端状态刷新

## 7.2 应用启动时的状态恢复

客户端启动后，需要先完成一次“我现在是谁”的探测，再渲染受保护区域。

### 推荐流程

1. 客户端读取本地持久化会话
2. 若存在凭证，则向认证系统确认其有效性
3. 将确认结果写入统一认证状态
4. 标记认证初始化完成
5. 再决定进入登录区还是主应用区

### 为什么必须有初始化阶段

如果应用一启动就立即按默认未登录状态渲染，很容易出现：

- 先看到登录页，再瞬间跳主应用
- 用户已经登录，但页面误判为未登录
- 启动瞬间进入错误路由

## 7.3 会话刷新与续期

长会话产品通常需要刷新机制。

### 设计要求

- 访问令牌过期前自动刷新
- 刷新失败时，状态应明确退回未登录或待重试
- 刷新期间不要重复弹出多个错误提示

### 推荐行为

- 前台活跃时自动维持会话
- 后台挂起时可暂停低优先级刷新
- 恢复前台时重新校验会话可用性

## 7.4 退出登录

退出登录必须被视为一条完整业务流程，而不是简单清空一个布尔值。

### 基本流程

1. 用户触发退出
2. 客户端调用服务端退出接口或撤销当前会话
3. 无论接口结果如何，本地敏感凭证都应被清理
4. 统一认证状态被重置
5. 导航栈回到认证入口

### 设计要求

- 退出后不能保留可继续访问受保护区域的本地状态
- 退出后刷新页面或重启应用，不应恢复旧会话

## 8. 验证与风控设计

## 8.1 发送验证码限制

发送验证码是最容易被滥用的动作之一。

### 设计要求

- 单邮箱短时间内限制发送频率
- 单设备短时间内限制发送频率
- 必要时增加图形验证、人机验证或设备指纹校验

### 用户反馈要求

- 给出剩余冷却时间
- 明确提示“请稍后重试”
- 不要用含糊错误文案让用户反复点击

## 8.2 验证码校验限制

### 设计要求

- 验证码应设置过期时间
- 同一验证码的尝试次数应有限制
- 连续失败达到阈值后，应进入短暂冷却期

## 8.3 密码尝试限制

### 设计要求

- 密码登录连续失败后进入冷却
- 高风险情况下可要求额外验证
- 不返回“用户不存在”或“密码错误”这类可被枚举利用的区别性信息

## 8.4 密码强度要求

至少应根据产品风险等级决定密码策略。

### 推荐基线

- 最小长度要求
- 不能过于简单
- 建议包含多种字符类型
- 禁止与邮箱、用户名等强相关内容完全一致

## 9. 页面与导航设计原则

## 9.1 不要把导航切换完全托付给异步副作用

登录成功后，如果页面跳转依赖多个地方各自监听状态变化，容易出现重复跳转、状态竞争和页面闪烁。

### 推荐原则

- 认证业务负责更新统一状态
- 页面根据明确的业务完成点进行一次性跳转
- 启动期准入由全局认证状态控制
- 业务流中的“下一步页面”由当前场景明确控制

## 9.2 中间状态必须有专属页面

以下状态不应被强行塞进“未登录”或“已登录”两个盒子里：

- 已验证但未设置密码
- 忘记密码验证成功，等待重置
- 会话恢复中

这些都应有明确页面和跳转规则。

## 10. 错误处理设计

认证系统应把错误分成以下几类：

- 输入错误：邮箱格式错误、验证码位数错误、密码不满足要求
- 业务错误：验证码错误、验证码过期、密码错误、会话失效
- 风控错误：发送过快、尝试过多、操作受限
- 网络错误：超时、断网、服务不可达
- 系统错误：未知异常、服务端故障

### 设计要求

- 输入错误在表单层即时反馈
- 业务错误给出清晰文案
- 风控错误必须说明等待成本
- 网络错误允许用户重试
- 系统错误应被记录并可观测

## 11. 数据与接口边界设计

虽然本文档不绑定接口细节，但建议不同项目至少保持以下边界清晰。

### 11.1 客户端职责

- 采集登录输入
- 提交认证请求
- 维护统一认证状态
- 执行页面准入控制
- 在会话变化时同步更新界面

### 11.2 服务端职责

- 校验身份凭证
- 控制验证码签发与校验
- 签发、刷新、撤销会话
- 管控风控规则
- 返回账户完成状态或可推导该状态的必要字段

### 11.3 建议返回的信息

认证成功后的响应建议至少能推导出：

- 当前用户身份
- 当前会话是否有效
- 账户是否已完成初始化
- 是否需要进入下一步引导

## 12. 可观测性设计

认证系统必须可监控，否则出了问题只能靠猜。

### 建议记录的事件

- 登录尝试
- 登录成功
- 登录失败
- 验证码发送
- 验证码验证成功/失败
- 密码设置成功/失败
- 会话恢复成功/失败
- 会话刷新成功/失败
- 退出登录

### 建议关注的指标

- 登录成功率
- 验证码发送成功率
- 验证码验证成功率
- 重置密码完成率
- 会话恢复失败率
- 因风控触发的拦截比例

## 13. 测试要求

一套可复用认证设计必须能被系统测试。

### 13.1 功能测试

- 密码登录成功
- 密码登录失败
- 验证码发送成功
- 验证码发送过快被限制
- 验证码验证成功
- 验证码验证失败
- 新用户首次验证后进入设置密码
- 忘记密码后成功重置
- 退出登录后回到认证入口

### 13.2 状态测试

- 应用启动时会话恢复
- 会话失效后正确退回未登录
- 已验证未完成初始化时不能进入主应用
- 完整登录后刷新页面仍保持正确状态

### 13.3 边界测试

- 重复点击登录按钮
- 重复提交验证码
- 登录过程中网络中断
- 会话刷新失败
- 多端登录导致本地会话失效

## 14. 推荐的最小可用方案

如果一个新项目资源有限，推荐先落以下最小闭环：

- 邮箱 + 密码登录
- 邮箱验证码登录
- 忘记密码
- 已验证但未设置密码的中间状态
- 启动时会话恢复
- 退出登录
- 基础冷却与错误提示

这样已经足够覆盖多数消费级产品的认证需求。

## 15. 未来扩展方向

在不推翻本设计的前提下，可以继续扩展：

- 手机号验证码登录
- 第三方 OAuth 登录
- 多因素认证
- 设备级信任管理
- 风险设备识别
- 登录审批与异常提醒
- 更细粒度的账户初始化流程

## 16. 结论

一套真正稳定的认证系统，不应该只回答“用户有没有登录”，还必须回答：

- 用户是否已经通过身份验证
- 用户是否已经完成账户初始化
- 当前会话是否仍然有效
- 客户端是否已经把认证状态同步完成

只要把这四个问题拆清楚，登录、验证码、忘记密码、首次设置密码、会话恢复和退出登录这些流程就都能落在同一套统一设计里。对于其他项目，最有价值的不是照搬页面形态，而是复用这套状态分层、准入规则和流程边界。
